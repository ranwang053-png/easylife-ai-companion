# Easylife AI 能力矩阵与 Provider 规划

最后更新：2026-06-27

## 1. 使用原则

- Flutter 客户端不得保存任何 AI API Key，也不得直接调用需要私密凭证的模型接口。
- 所有真实 AI 能力统一通过自建后端代理调用，后端负责密钥、限流、脱敏、审计、重试和供应商切换。
- 情绪分析与数据保存分离：AI 分析不自动保存；用户确认记录后，才保存情绪日记和长期记忆。
- 开发日志、访问日志和追踪不得记录手机号、短信验证码、Token、完整情绪原文、长期记忆正文或用户画像。
- 正式 App 用户可见界面不得出现 Mock、测试、示例、原型等实现态字眼。
- 模型只生成需要语义判断或自然语言表达的内容；日期、统计、求和、排序、
  状态映射和 UI 展示字段由后端或前端确定性生成。
- 用户画像上下文默认由后端按用户设置、能力用途和长期记忆标签拼装，不在每次
  AI 请求前额外调用模型生成摘要。

## 2. 总体架构

```text
Flutter UI
  -> Easylife Backend API
    -> AI Gateway
      -> Text Provider
      -> Vision Provider
      -> Image Generation Provider
      -> Search / Nutrition Provider
      -> Segmentation Provider
```

推荐先实现统一的 `AI Gateway`，每个能力背后可以选择不同供应商，不要求整个 App 绑定同一个供应商。

## 5. Prompt 与风格资产规划

伙伴形象生成建议同时使用两类资产：

| 类型 | 放在哪里 | 用途 | 是否必需 | 备注 |
| --- | --- | --- | --- | --- |
| Prompt 风格规则 | 后端配置或 Prompt 模板库 | 定义“干净轮廓、浅薄荷底座、克制陪伴感”等规则 | 必需 | 便于版本管理和快速迭代 |
| 风格参考图 | 对象存储 + 数据库元数据 | 作为 style reference 或人工质检基准 | 推荐 | 先准备 3-10 张最满意样例 |
| 禁忌清单 | Prompt 模板库 | 避免低龄化、贴纸化、夸张腮红、重阴影等 | 必需 | 与当前 UI 原则一致 |
| 生成结果 | 对象存储 | 用户确认后保存头像/形象 | 必需 | 未确认结果建议短期缓存 |

当前已归档 Prompt：

- 伙伴形象生成：`backend/prompts/pet_avatar_generation.v1.md`
- 每日运势生成：`backend/prompts/daily_fortune_generation.v1.md`
- 陪伴与情绪日记：`backend/prompts/companion_reply.v1.md`、
  `backend/prompts/emotion_journal_summary.v1.md`
- 饮食识别、分析、建议和回顾：`backend/prompts/food_vision_recognition.v1.md`、
  `backend/prompts/nutrition_analysis.v1.md`、
  `backend/prompts/diet_advice_planning.v1.md`、
  `backend/prompts/diet_review.v1.md`
- 长期记忆：`backend/prompts/long_term_memory_extraction.v1.md`
- 用户画像上下文：`backend/prompts/user_profile_summary.v1.md`，当前为后端规则拼装规范，
  不执行实时模型调用。

## 6. 接入顺序建议

1. 后端 AI Gateway 与环境变量管理。
2. 情绪分析真实 Provider，保持前端接口不变。
3. 日志脱敏、超时、限流和失败兜底。
4. 长期记忆提炼 Provider。
5. 深呼吸与轻音乐独立页面先做前端体验，再考虑 AI 推荐。
6. 伙伴形象生成的 Prompt、风格参考库和对象存储。
7. 饮食文字热量估算、照片识别、联网搜索和抠图分 Provider 接入。
