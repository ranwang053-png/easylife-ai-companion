# Easylife 项目记忆

本文件是本项目供 Codex 和其他开发 Agent 使用的长期记忆。开始任务前先阅读
本文件，再阅读与任务直接相关的代码。完成重要功能、架构调整或产品决策后，
同步更新本文件中已经失效的内容。

## 1. 产品定位

Easylife 是一个 Flutter 实现的 AI 生活陪伴 App MVP。产品希望通过温和、低压、
有陪伴感的交互，把用户画像、桌宠陪伴、情绪记录和饮食体重连接起来。

当前阶段快速完成上述四项核心能力，并在首页保留每日运势卡片；不加入灵感创作、
塔罗等独立扩展板块。架构应保持可迭代，但不要为尚未进入 MVP 的功能提前增加代码和维护成本。
不把 Mock 行为伪装成真实 AI、真实账号系统或持久化后端。

自 2026-06-14 起解除前端功能冻结。允许优化现有页面的布局、配色、动效和组件，
也允许在保持现有功能与主导航不变的前提下全面重新设计 UI。新增功能可以继续迭代，
但必须由用户当次明确提出，并同步更新产品范围、架构、测试和相关文档，不能在无明确
需求时自行扩张功能。
首发平台确定为 iOS；打包、真机验收、签名和 TestFlight 准备均以 iOS 为优先，
Android 打包环境暂缓，不作为当前内测上线阻塞项。

## 2. 当前实现状态

- 技术栈：Flutter、Dart、Material 3。
- 包名：`company_app`。
- App 入口：`lib/main.dart`。
- 启动流程：登录或注册 -> 用户基础信息 -> 主 App；宠物档案由用户之后按需创建。
- 主导航：`lib/pages/app_shell.dart`，包含首页、陪伴、饮食、我的。
- 首页聚焦桌宠互动、每日运势和今日记录状态。
- 陪伴模块支持文字或系统语音转写输入、Mock 分析、建议与历史展示；语音只负责转成
  文字，转写文本再通过 `AgentService` 分析。
- 伙伴状态显示在伙伴名称旁，并由最近一条已保存情绪日记映射生成；未记录时默认为
  “期待”。伙伴状态表达对用户状态的互动反应，例如疲惫映射为“担忧”、低落映射为
  “心疼”，超过 3 天没有新日记时显示“期待”。
- 情绪分析与情绪日记支持一个主标签和多个辅助标签；旧的单标签本地数据继续兼容。
- 陪伴页采用轻量多轮对话：发送后清空输入框并保留本轮气泡；水上体验保持朋友式陪伴，
  不在单轮聊天中强行展示报告式结构。用户主动点击“保存情绪日记”后，才将本轮用户发言
  整理为一条情绪日记，并提炼有限条本地长期记忆写入 `UserProfile.memoryNotes`。
  `AgentService` 后续分析会收到这些记忆；当前最多保留最近 12 条提炼结果，不长期保存
  全部逐字聊天记录。真实 AI 陪伴回复会同时收到低敏伙伴档案上下文，包括伙伴名称、
  性格标签、关系说明和性格摘要，用于轻微贴合伙伴口吻；不发送伙伴照片、生日或生成图
  URL。
- 情绪日记详情页使用温和卡片呈现整理结果，避免直接展示“摘要”“可能原因”“分析报告”
  等机械标题；允许适度 emoji 增强拟人感。
- 饮食模块支持体重记录、食物描述或图片占位、Mock 热量估算、手帐贴纸流程，以及“饮食回顾”
  页面。“饮食回顾”不做任意日期选择，保留日、周、月维度，并在每个维度下展示前一日/周/月
  数据总结、当日/周/月数据总结和下一日/周/月饮食建议；重点是 AI 复盘、结构洞察和调整建议，
  不是逐条流水账。收藏贴纸库和书本入口仍待产品确认，当前不要提前实现。
- 我的与设置模块承载用户画像、宠物档案及偏好编辑。
- 用户画像包含可上传和更换的用户头像；“我的”和用户画像页展示用户头像，
  伙伴档案、首页和陪伴页展示伙伴形象，二者不得混用。
- 系统偏好支持在本地编辑权限与隐私偏好，包括通知、定位、麦克风、相机相册、
  云端同步、长期记忆个性化和诊断日志；当前仅保存用户偏好，不等同于
  已调用真实 iOS 系统授权弹窗。
- 当前 AI 能力通过 `AgentService` 统一调用：未配置后端时使用 `MockAgentService`；
  配置 `EASYLIFE_API_BASE_URL` 后，陪伴回复、情绪日记整理、长期记忆提炼、旧情绪分析
  和伙伴形象生成优先走自建后端，失败时按本地优先策略回退到本地示例能力。
- 后端 OpenAI-compatible provider 支持 `OPENAI_API_KEY`、`OPENAI_BASE_URL` 和
  `OPENAI_MODEL`；`OPENAI_BASE_URL` 未配置时默认使用 OpenAI 官方 `/v1` 地址，配置为
  `https://aihubmix.com` 这类根地址时会自动补 `/v1`。`OPENAI_MODEL` 作为 OpenAI
  文本能力的统一模型覆盖项，不影响图像生成模型。
- 生产入口使用 `shared_preferences` 保存用户、宠物、情绪、饮食、体重和引导状态；
  Widget 测试继续使用内存实现隔离数据。
- 生产认证使用 `HttpAuthService` 对接 `EASYLIFE_API_BASE_URL`，访问令牌和刷新令牌
  通过系统安全存储保存；启动时恢复会话，访问令牌临近过期时自动刷新，刷新失败返回
  登录页。固定示例认证仅用于测试和本地演示。
- Node.js 后端已实现 PostgreSQL 真实认证：短信 challenge、自动注册、15 分钟
  Access Token、30 天一次性轮换 Refresh Token、数据库会话校验、旧刷新令牌重放
  撤销和当前设备幂等退出。生产启动必须配置数据库、独立密码学密钥和 HTTPS 短信网关；
  固定 Token 仅保留为开发与契约测试夹具。
- 本地 Web 预览可通过编译期开关 `EASYLIFE_PREVIEW_AUTH_BYPASS=true` 临时绕过登录进入

- AI Prompt 已完成首轮成本瘦身：模型只生成需要语义判断或自然语言表达的字段；

  由后端或前端确定性生成。
- 多轮陪伴每轮通过 `POST /v1/companion/reply` 生成自然回复、可选情绪标签、风险等级和
  可选服务建议；情绪强度、可能原因和结构化日记只在用户主动保存时，通过
  `POST /v1/emotion-journals/summarize` 根据整轮对话统一生成。现有冻结的
  `/v1/emotion/analyze` 继续保持兼容。
- 长期记忆提炼通过 `POST /v1/memories/extract` 从用户已确认保存的内容中提出候选；
  接口不直接修改记忆列表，客户端仍执行本地优先保存和同步。
- 用户画像上下文不在每次 AI 调用前使用模型实时总结；后端按用户设置、能力用途和
  低敏长期记忆标签进行规则筛选、去重和限长。只有长期记忆超过阈值时才允许异步压缩。
- 长期记忆不仅来自用户确认保存后的 AI 提炼，用户也可以在画像页进入独立管理页，
  主动查看、添加、修改和删除记忆；当前本地最多保留 12 条，修改后立即持久化并用于
  后续 AI 调用。管理页对自动提炼记忆只展示冒号前的整理认知，不展示冒号后的原始
  日记摘要；底层 `memoryNotes` 和 AI 调用逻辑保持兼容。
- 每日运势整体解读限制为 180-260 字；整体分数不再单独生成重复解释。
- 饮食统计、热量与营养素合计、最常吃、最高摄入等指标由后端计算；饮食模型只负责
  单项识别与估算、结构观察和少量自然语言建议。
- 第一阶段云端同步用户画像、宠物档案、情绪日记和独立长期记忆；饮食和体重留到
  第二阶段。用户数据本地优先，联网后自动增量同步。
- 开发日志、访问日志和追踪不得记录手机号、短信验证码、Token、完整情绪原文、
  长期记忆正文或用户画像。
- 首发平台为 iOS；当前 Mac 已安装 Flutter，但只有 Apple Command Line Tools，
  尚未安装完整 Xcode 和 CocoaPods。当前 macOS 14.6 不能运行满足 2026 年
  App Store Connect 上传要求的 Xcode 26，需先升级 macOS。
- 当前前端允许持续优化和全面重新设计 UI；未明确新增功能时，应保持现有功能与主导航
  不变。新增功能需经用户明确确认并同步调整项目规划。
- 当前 UI 采用“薄荷清新 80% + 极简轻奢 20%”视觉：大面积奶油白、极浅薄荷与雾灰，
  薄荷绿用于按钮、标签、进度与选中态，香槟金仅用于幸运信息等小面积点缀。
- 主要卡片统一为 28px 圆角、低透明度浅薄荷边框和轻微绿色调阴影；输入框为 16px
  圆角，主要按钮使用深墨绿色胶囊形，并保留 Cupertino 页面转场。
- 布局分为三档：手机使用底部导航；平板增加内容留白并在首页使用双栏；900px 及以上
  使用左侧导航和受控内容宽度。功能入口和四项主导航保持一致。
- 每日运势是首页固定卡片，包含整体运势、幸运色、幸运食物、幸运数字、幸运花、
  事业/财富/爱情/人际分数、建议和避免；灵感创作与塔罗不属于当前 MVP。

`README.md` 和 `DELIVERY.md` 可用于了解背景，但可能晚于代码更新。发生冲突时，
以当前代码、测试和 `pubspec.yaml` 为事实来源，并修正文档。

## 3. 架构边界

页面只负责展示、输入和流程编排，不直接包含 Prompt、模型供应商逻辑或 API Key。

```text
Flutter Page
    -> AgentService
        -> MockAgentService（当前）
        -> HttpAgentService（未来）
            -> 自建 Node.js / Express API
                -> AI Model + Database
```

必须遵守：

- 可优化现有页面的布局、配色、动效和组件，也可全面重新设计 UI。
- 单纯 UI 重设计默认保持现有功能和主导航不变，不借视觉改版暗中改变产品范围。
- 可以新增页面、导航入口或业务能力，但必须由用户当次明确提出，并同步更新产品规划、
  数据模型、service/repository 边界、测试和文档。
- 本地存储和单一情绪分析 AI 接入优先于其他远程能力、图片上传和新功能开发。
- AI 业务统一通过 `lib/services/agent_service.dart` 的 `AgentService` 抽象。
- 新增 AI 能力时，先扩展结构化 model 和 service 接口，再接入页面。
- Flutter 客户端不得保存模型 API Key，也不得直接调用需要私密凭证的模型接口。
- 未来后端优先自建 Node.js / Express；QClaw 仅可参考 Prompt、工作流和输出格式，
  不作为正式运行时依赖。
- 后端数据库使用 PostgreSQL，迁移文件是数据库结构的事实来源；生产环境不得依赖
  ORM 自动同步 Schema。
- 所有业务数据查询和修改必须使用访问令牌中的用户 ID 限定数据归属，不信任客户端
  提交的用户 ID。
- 同步写入使用用户范围内唯一的 `mutationId` 保证幂等，冲突以服务端版本号判断，
  删除使用墓碑；不得使用设备时间直接决定冲突胜负。
- 字段、错误码、鉴权、限流、保存时机或隐私边界发生变化时，必须先由产品、前端和
  后端共同确认，再更新 `contracts/openapi.yaml`、示例和 `CHANGELOG.md`。禁止先改
  实现再让契约追认。
- 前后端联调前必须分别通过自己的契约测试；联调清单通过后才能进入 TestFlight 验收。
- 用户注销需要二次短信验证，立即撤销全部会话并停止同步，随后异步彻底删除数据库、
  文件、缓存和第三方供应商中的用户数据。
- 数据读写通过 service 或 repository 隔离，避免页面直接绑定具体存储方案。
- 将内存 Mock 替换为 HTTP、Hive、SQLite 或云端实现时，尽量保持页面调用契约稳定。

## 4. 代码地图

- `lib/main.dart`：应用启动、依赖实例和启动阶段状态机。

- `contracts/CONTRIBUTING.md`：契约变更审批和兼容规则。
- `docs/PARALLEL_DEVELOPMENT.md`：前后端责任边界、契约测试、联调和 TestFlight 门禁。

- `backend/database/README.md`：手机号保护、同步和日志安全约束。
- `backend/database/ACCOUNT_DELETION.md`：账号彻底删除运行手册。
- `backend/server/src/migrate.ts`：迁移版本登记、锁和校验和入口。
- `backend/server/src/worker.ts`：账号注销任务和认证过期数据清理 worker 入口。
- `backend/prompts/`：后端 AI Prompt 模板库，按能力和版本号保存，不存放 API Key。
  当前包含伙伴回复、情绪日记整理、每日运势、伙伴形象生成、食物识别、营养分析、

  读取前 3 张，需重启后端生效，图片文件默认不纳入 git。

新增文件时沿用现有职责划分，不把大型业务逻辑堆进 `main.dart` 或页面 `build`
方法。

## 5. 产品与 UI 原则

- 中文是当前主要界面语言。
- 视觉基调应温柔、简洁、低压；允许调整现有配色、布局、动效和组件体系。
- 现行设计系统以 iOS 体验为起点并适配平板和桌面：整体高级、干净、精致、温柔，
  保持低信息密度、清晰字号层级和充足留白。
- UI 视觉优化默认不得修改入口位置、信息顺序、导航结构或交互逻辑；仅调整颜色、字体、
  间距、圆角、边框、阴影、图标容器和局部材质表现。
- 正式 App 的用户可见界面不得出现 “Mock”“测试”“示例”“原型”等实现态或测试态字眼；
  这些词只允许出现在代码、测试和内部文档中。
- 视觉应有轻微材质与层次，但避免重阴影、泛滥玻璃效果、商务仪表盘、儿童化插画和
  传统健康管理工具式的数据堆叠。
- 卡片默认 28px 圆角，输入框默认 16px 圆角，主要按钮使用胶囊形；新增界面应沿用
  这一形状规则，除非后续明确调整整套设计系统。
- 宠物形象保持克制陪伴感：使用干净轮廓、浅薄荷底座和柔和阴影，避免明显腮红、贴纸化
  装饰、夸张表情和低龄文案。
- 页面内容必须设置合理最大宽度；不得把手机单列页面直接拉伸到桌面全宽。新增主页面
  应同时检查 390px、820px 和 1280px 三档尺寸。
- UI 局部优化时优先复用 `AppColors`、`AppTheme`、`SoftCard`、`PageHeader` 等既有
  设计元素；全面重设计时可有计划地替换，但应保持设计系统一致并同步迁移调用方。
- AI 输出需要明确、结构化、可操作，避免夸大能力或制造确定性。
- AI 输出字段遵循最小化原则：新增字段前必须确认实际消费方；调用方已持有的数据、
  可由代码计算的数据和纯 UI 展示字段不得要求模型重复生成。
- 情绪和健康内容不能替代专业医疗、心理或营养建议。
- 宠物或人物照片用于生成形象前应保留清楚的用户授权说明；伙伴形象生成目标是透明背景
  全身卡通人物/宠物抠图，即使上传半身照也应由模型补全为完整站立角色。
- 新流程需要处理 loading、空状态、错误、取消、重复点击和 Widget 生命周期。
- 异步回调更新 UI 前检查 `mounted`，避免页面退出后的 `setState`。

## 6. 开发约定

- 遵循现有 Dart 风格和 `analysis_options.yaml`。
- 优先做范围明确的小改动，不顺手重构无关模块。
- 结构化数据使用明确的 model，不在页面之间传递松散 Map。
- 使用构造函数注入 service，保持页面可测试。
- Mock 与正式实现使用相同接口，Mock 文案和延迟应稳定、可预测。
- 新增用户可见功能时补充或更新 Widget 测试。
- 修改模型或 service 契约时，检查所有实现、调用方和测试。
- 不提交生成目录，例如 `build/`；不要手动编辑 Flutter 生成文件，除非平台配置
  确实要求。
- 未经明确要求，不覆盖或回退用户已有改动。

## 8. 当前已知方向

- 持续优化现有 UI；全面重设计时默认保持现有功能与主导航不变。
- 可按用户明确需求新增功能，并同步更新项目规划、架构、测试和文档。
- 本地持久化基础设施和核心数据迁移已完成，继续验证真机重启恢复和数据损坏回退。
- 情绪分析 `HttpAgentService` 边界已完成，下一步由自建后端实现 API 契约。
- PostgreSQL 认证 Schema 与 Node.js / Express 真实认证已实现，下一步部署持久化测试
  数据库、接入选定短信供应商，并继续实现同步事务。
- 固定响应契约测试和真实认证数据库集成测试已通过，下一步进行真实短信与持久化
  PostgreSQL 联调，再逐项替换同步和 AI provider；完成真实依赖联调前不得进入
  TestFlight。
- iOS 为首发平台，需优先补齐完整 Xcode、CocoaPods、Apple 签名和 TestFlight 环境。
- Android 发布与 Android SDK 安装暂缓，不阻塞当前 iOS 内测。
- 其他真实 AI、图片上传和新功能是否推进，以用户当次明确需求和更新后的项目规划为准。
- 实现真实鉴权、错误处理、隐私与数据删除机制。
- 扩大 repository、序列化、迁移和关键用户流程的测试覆盖。

这些是方向，不代表已经承诺的具体排期。MVP 稳定后可以继续迭代新板块，但实现前
必须以用户当次需求为准，不提前恢复灵感创作或塔罗；首页每日运势卡片属于当前范围。
